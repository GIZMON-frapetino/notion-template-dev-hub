# T-03 運用方法

> 正本: 本リポジトリ（GitHub）。Notion 上のページは配布・レビュー用。

### 簡単な説明

クライアント管理から、商談→案件→コミュニケーション履歴までをリレーションで一本化した、フリーランス向けのCRMテンプレートです。次アクションと期限が自然に溜まり、案件の進捗・売上見込み・やり取り履歴が、探さなくても見える状態になります。

### 詳細な説明

「顧客情報」「商談の判断」「案件の実行」「やり取り履歴」が別々に散らばると、

- 受注確度が上がらない
- 案件の品質がブレる
- 請求やフォローが抜ける
- 引き継ぎが難しい

といった問題が起きやすくなります。

このテンプレートは、以下の流れを **Notion上で一気通貫** で回すために設計しています。

- **クライアントDB**：顧客マスター（企業情報・契約形態・LTVなど）
- **商談パイプラインDB**：判断と決定を先に整理し、フェーズを前に進める
- **案件DB**：受注後の実行管理（請求単位で管理し、本文に成果物・検収・意思決定を残す）
- **コミュニケーションログDB**：やり取りを時系列で資産化し、次回確認日で漏れを防ぐ

---

## セットアップ手順（最初にやること）

1. このテンプレートを自分のワークスペースに **複製** する
2. **クライアントDB** に、既存のクライアントを登録する（まずは3〜10件でOK）
3. 進行中の案件があれば **案件DB** に登録し、クライアントと紐づける
4. 商談中の案件があれば **商談パイプラインDB** に登録し、クライアントと紐づける
5. 直近のやり取りを **コミュニケーションログDB** に1〜3件だけ登録する

---

## 運用ルール（迷わないための最小ルール）

### 1. クライアントDB（起点）

- まずは **企業名・担当者・業界・契約形態** だけ埋めればOK
- LTVは「案件完了/請求が確定したタイミング」で更新

### 2. 商談パイプラインDB（判断を前に進める）

- 商談の目的は「会話を増やす」ではなく **判断を前に進める** こと
- 商談ページ本文に、次を必ず残す
  - STEP1：MECEヒアリング要点
  - STEP2：本音（潜在不安）の想定
  - STEP3：勝ち筋（受注条件）
  - 決定事項 / 未決事項
- **次アクション期限** は必須（放置防止）
- 受注が確定したら、案件DBにレコードを作成し、商談と紐づける

### 3. 案件DB（品質の安定化）

- 案件は基本 **請求単位** で管理する
- プロパティは「一覧で追う」ために最小限に。
- 長文になりやすい内容は本文へ集約する
  - 作業内訳（統合）
  - 成果物定義
  - 検収条件
  - 意思決定ログ
  - リスク・懸念

### 4. コミュニケーションログDB（履歴を資産化）

- やり取り直後に **30秒で要点だけ** 記録する
- プロパティは、検索と時系列管理に必要な項目だけ入れる
  - タイトル / 日時 / 種別 / 次回確認日 / クライアント / 案件（必要なら商談）
- 本文は「文脈と次の打ち手」だけ残せばOK

## 振り返りサイクル（おすすめ）

| 頻度 | 見るビュー / ページ | やること | 所要時間 |
| --- | --- | --- | --- |
| 毎日 | コミュニケーションログDB（フォローアップ） | 次回確認日が来ているものを処理 | 1〜3分 |
| 週次 | 商談パイプラインDB（パイプライン） | フェーズと次アクション期限を更新 | 5分 |
| 週次 | 案件DB（優先度別ボード / タイムライン） | 今週の優先案件、納期、入金予定を確認 | 5〜10分 |

## Tips：AIで“入力作業”を減らす（おすすめ運用）

このテンプレートは、人が毎回ゼロから入力する前提ではなく、**AIにプロパティと本文を埋めてもらい、人は最終確認だけする**運用に寄せると回りやすくなります。

### 1. 会話ログ / 議事録 → テンプレ照合 → DB登録（半自動）

やりたいことはシンプルで、

- 会話ログの要点（サマリー）
- 議事録
- メール

などのテキストから、テンプレ（案件/商談/コミュニケーションログの本文テンプレ）に合わせて **必要項目を抽出**し、DBに転記します。

**想定フロー（例）**

1. 会話ログサマリーや議事録を用意する（コピー/貼り付けでOK）
2. AIがテンプレに照合して「プロパティ候補」と「本文ドラフト」を生成する
3. 人は「次アクション」「期限」「紐づけ（クライアント/商談/案件）」だけ確認して確定
4. API / MCP を使う場合は、AIが **JSON** 形式で出力し、そのままDBに投入する

[T-03 API連携仕様書](https://www.notion.so/T-03-API-2b9ac2dff4364a21bcdff7e741e87c4b?pvs=21)（GitHub正本: [api-integration.md](./api-integration.md)）

**コツ**

- まずは **コミュニケーションログDB** を入口にして、後から商談・案件へリレーションで繋げるとスムーズです。
- 長文（背景、意思決定、懸念）は本文へ。
  - プロパティは「検索・期限・紐づけ」中心に絞ると破綻しません。

### 2. 最小必須だけ“先に”埋め、残りは後でAIに補完させる

忙しい時は、以下だけ先に入れれば十分です。

- **次アクション**
- **期限（次アクション期限 / 次回確認日）**
- **クライアント**
- （可能なら）**商談 / 案件**

残りは、週次レビュー時にAIへ「この週のログを要約して不足を補完して」と依頼して埋め直せます。

### 3. 週次レビューを“AIの得意領域”に寄せる

週次の見直しでは、AIに次をやらせると効果が出ます。

- フェーズ更新の提案（商談パイプラインDB）
- 未決事項の抽出（商談本文）
- リスクの言語化（案件本文のリスク・懸念）
- 次アクション候補の提案（期限付き）

### 4. テンプレを“入力インターフェース”として使う

テンプレの価値は文章の見栄えではなく、

**AIが読み取れる構造（MECE）** を用意できることです。

- 本文テンプレは「AIに書かせる前提」で、見出しと箇条書きを固定
- プロパティは「後で集計・検索する軸」に限定

---

## 運用負荷を下げるコツ

- 記録は **完璧を目指さない**（8割でOK）
- 迷ったら「次アクション」と「期限」だけを先に入れる
- 長文は本文へ。DBの列は増やしすぎない
- 2週間運用して負荷が高ければ、入力項目を減らして再設計する

---

## 2週間後の判断基準（継続できる形にチューニング）

- 記録が継続できている → 本格運用へ
- 記録が途切れがち → 必須プロパティを最小化して再検証
- そもそも負荷が大きい → 本文中心の運用に寄せ、DB入力を減らす

### よくある質問（FAQ）

- **Q. どのDBから入力すればいい？**
  - A. まずはコミュニケーションログDBか商談DBからでOKです。迷う場合は「次アクション」「期限」「クライアント」だけ先に入れます。
- **Q. 長文が増えてDBが重くならない？**
  - A. 長文は本文へ集約し、プロパティは検索・期限・紐づけ中心にする運用を推奨します。
- **Q. 商談が受注したら何をする？**
  - A. 案件DBにレコードを作成し、商談DBの「案件」リレーションで紐づけます。
- **Q. 複数請求（月次など）の案件は？**
  - A. 親（請求単位）を作り、必要なら子案件で内訳を持ちます。請求ステータスと入金予定日を更新してください。
